# Model Strategy for Standards LLM Wiki

本文描述本项目中不同类型 AI 模型的职责边界、选型标准与推荐工作流。

本项目面向标准法规知识库建设，核心目标不是简单摘要 PDF，而是构建可追溯、可维护、可用于 AI 问答的结构化 Markdown Wiki。因此模型分工必须优先考虑：

- 事实准确性；
- 条款与来源可追溯；
- 跨标准关系理解；
- 版本差异判断；
- 成本与吞吐量；
- 异步任务的可重试性。

## 1. 模型能力要求

标准法规知识库对模型能力的要求主要包括：

### 1.1 长文档理解

模型需要处理标准全文、附录、表格、规范性引用文件、修改说明、实施公告等长文本材料。

重点能力：

- 维持条款编号稳定；
- 不混淆章节层级；
- 能从长文档中定位局部事实；
- 能识别正文、附录、注释、表格之间的关系。

### 1.2 精确事实抽取

模型需要抽取：

- 标准号；
- 标准名称；
- 发布日期；
- 实施日期；
- 替代标准；
- 被替代标准；
- 适用范围；
- 规范性引用文件；
- 条款编号；
- 适用对象；
- 过渡期说明。

此类信息必须忠实于来源，不能补全、猜测或改写成未经验证的结论。

### 1.3 跨文档关联

典型问题：

> 涉及到座椅的都有哪些标准？

这要求模型能把多个标准中的相关主题关联起来，例如：

- 座椅；
- 座位；
- 座椅布置；
- 座椅强度；
- 安全带固定点；
- 客车座椅；
- 校车座椅；
- 检验项目。

### 1.4 版本差异分析

典型问题：

> 新版 GB 7258 相比旧版，更新了哪些地方？

模型需要完成：

- 章节对齐；
- 条款对齐；
- 新增、删除、修改识别；
- 语义变化判断；
- 影响范围归纳；
- 重大变化优先级排序。

### 1.5 保守回答与不确定性控制

标准法规场景中，模型必须能够明确表达：

- 当前资料未找到；
- 需要核查官方公告；
- 该结论仅基于某一版本原文；
- 是否存在过渡期需要结合实施公告或修改单确认。

最终答案不应把未核实信息表达成确定结论。

## 2. 模型分层

本项目建议将模型分为四类：

```text
可靠即时高推理模型      推理能力强、稳定、适合关键判断
可靠即时中等能力模型    稳定、响应快、适合结构化初抽
不稳定高能力后台模型    推理能力强但稳定性不足，适合异步交叉审查
不稳定中等能力后台模型  能力一般或输出不稳定，适合批量候选生成
```

总体原则：

```text
高推理模型负责判断、归纳、审核、最终回答。
中等能力模型负责批量、低风险、候选生成。
不稳定后台模型负责异步后台处理，不进入实时问答链路。
```

## 3. 可靠即时高推理模型

### 3.1 定位

可靠即时高推理模型是项目中的主判断模型与最终出口模型。

适合承担：

- 知识库架构设计；
- Schema 和页面模板设计；
- 首批样本标准精加工；
- 条款解释；
- 跨标准主题页整理；
- 版本差异分析；
- 实施日期、过渡期、替代关系判断；
- 后台模型产物审核；
- 最终用户问答。

### 3.2 适合任务

| 任务 | 是否推荐 | 说明 |
|---|---:|---|
| 目录结构设计 | 推荐 | 需要抽象和长期可维护性判断 |
| `SCHEMA.md` 设计 | 推荐 | 需要定义长期约束 |
| 标准主页面精加工 | 推荐 | 首批样本尤其应使用高推理模型 |
| 条款白话解释 | 推荐 | 需要保守解释，避免扩大适用范围 |
| 跨标准主题页 | 推荐 | 需要去重、合并同义概念、判断关联强弱 |
| 版本对比 | 强烈推荐 | 中等能力模型容易漏语义变化 |
| 最终问答 | 强烈推荐 | 面向用户的结论应由高推理模型输出 |
| 大批量低风险预处理 | 不优先 | 成本较高，适合交给中等能力模型或后台模型 |

### 3.3 使用原则

- 首批 3-5 个标准应主要用可靠即时高推理模型打样。
- 正式知识库页面进入 `standards/`、`topics/`、`clauses/`、`comparisons/` 前，应由可靠即时高推理模型或人工审核。
- 对法规结论、适用范围、过渡期、版本变化等高风险问题，优先使用可靠即时高推理模型。

## 4. 可靠即时中等能力模型

### 4.1 定位

可靠即时中等能力模型适合作为快速生产力模型，承担低风险、结构化、可校验的任务。

适合承担：

- 元数据初抽；
- 章节目录抽取；
- 规范性引用文件初抽；
- 主题标签初筛；
- 条款候选召回；
- 简单摘要初稿；
- 索引初稿生成；
- 用户等待时的轻量辅助检索。

### 4.2 适合任务

| 任务 | 是否推荐 | 说明 |
|---|---:|---|
| 标准号、标题、发布日期抽取 | 推荐 | 结果必须可回源校验 |
| 章节目录抽取 | 推荐 | 可由规则和人工快速检查 |
| 规范性引用文件列表 | 推荐 | 适合正则辅助校验 |
| 主题关键词召回 | 推荐 | 重点在召回，不做最终判断 |
| 条款候选列表 | 推荐 | 输出候选，交给高推理模型筛选 |
| 每章一句话摘要 | 可用 | 作为草稿，不直接入正式页 |
| 版本影响分析 | 不推荐 | 容易漏掉隐含语义变化 |
| 最终法规结论 | 不推荐 | 风险较高 |

### 4.3 使用原则

- 输出应进入 `_drafts/`、`_candidates/` 或 `_reviews/`。
- 不直接写入正式 `standards/`、`topics/`、`clauses/`、`indexes/`。
- 抽取字段必须带来源片段、章节或条款号。
- 涉及法律/标准结论时必须交由可靠即时高推理模型或人工复核。

## 5. 不稳定后台模型

### 5.1 定位

不稳定后台模型适合作为异步后台 worker。它们不适合实时问答，也不应直接产出最终结论。

最佳定位：

```text
异步、可重试、可校验的候选生成器。
```

### 5.2 不稳定高能力后台模型

如果模型推理能力强但稳定性不足、响应不可预测或只适合后台运行，则适合承担：

- 夜间深度版本对比草稿；
- 跨标准主题综述初稿；
- 复杂关系图谱候选；
- 长文档深度摘要；
- 多标准合并分析初稿；
- 大批量标准页面草稿。

输出建议路径：

```text
_drafts/standards/
_drafts/topics/
_drafts/comparisons/
_reviews/background-high-capability-model/
```

### 5.3 不稳定中等能力后台模型

如果模型能力一般、稳定性不足或响应不可预测，则适合承担：

- 关键词召回；
- 条款候选扫描；
- 标准号识别；
- 章节切分；
- 主题标签初筛；
- 可能问题列表生成；
- 规范性引用文件候选抽取。

输出建议路径：

```text
_candidates/topic-clauses/
_candidates/cited-standards/
_drafts/questions/
_reviews/metadata-votes/
```

### 5.4 使用原则

不稳定后台模型必须满足以下隔离原则：

1. 不参与用户实时问答链路。
2. 不直接写入正式知识库目录。
3. 所有输出必须包含来源文件、章节/条款号、原文片段和置信度。
4. 任务失败可以重试，失败不阻塞主流程。
5. 结论性内容必须由可靠即时高推理模型或人工审核。

## 6. 推荐异步目录

为了承接后台模型和中等能力模型的输出，建议增加以下目录：

```text
_drafts/                 # 模型生成的草稿，未审核
_candidates/             # 候选条款、候选主题、候选引用
_reviews/                # 多模型交叉检查结果、审核记录
_jobs/                   # 异步任务描述和任务运行结果
```

推荐结构：

```text
standards-wiki/
├── _drafts/
│   ├── standards/
│   ├── topics/
│   ├── clauses/
│   ├── comparisons/
│   └── questions/
│
├── _candidates/
│   ├── topic-clauses/
│   ├── cited-standards/
│   ├── metadata/
│   └── topic-tags/
│
├── _reviews/
│   ├── metadata-votes/
│   ├── contradiction-checks/
│   ├── source-checks/
│   └── final-review/
│
└── _jobs/
    ├── pending/
    ├── running/
    ├── completed/
    └── failed/
```

## 7. 推荐工作流

### 7.1 入库流水线

```text
原始 PDF / 官方网页 / 法规公告
        ↓
OCR / PDF 转 Markdown
        ↓
不稳定后台模型池
- 批量摘要
- 候选条款召回
- 主题标签
- 元数据初抽
- 多模型交叉检查
        ↓
可靠即时中等能力模型
- 即时轻量抽取
- 快速补全索引
- 用户等待时的辅助检索
        ↓
可靠即时高推理模型
- 审核
- 归纳
- 版本对比
- 跨标准判断
- 最终回答
        ↓
正式 Markdown Wiki
```

### 7.2 白天即时链路

```text
用户问题
  → 检索正式知识库
  → 可靠即时高推理模型 综合回答
```

简单问题可选：

```text
用户问题
  → 检索正式知识库
  → 可靠即时中等能力模型初答
  → 可靠即时高推理模型必要时兜底或审核
```

### 7.3 夜间异步链路

```text
新增 sources/*.md
  → 不稳定后台模型批处理
  → 产出 _drafts/、_candidates/、_reviews/
  → 可靠即时高推理模型定期审核
  → 正式入库
```

## 8. 多模型交叉检查

对于客观字段，可使用多个模型交叉检查：

- 发布日期；
- 实施日期；
- 替代标准；
- 被替代标准；
- 规范性引用文件；
- 主题标签；
- 过渡期候选。

推荐规则：

```text
3/3 一致 → 可进入待审核
2/3 一致 → 可靠即时高推理模型复核
1/3 或冲突 → 人工/官方来源核查
```

交叉检查输出建议放入：

```text
_reviews/metadata-votes/
_reviews/contradiction-checks/
```

## 9. 模型选型标准

选择模型时应考虑以下维度：

| 维度 | 说明 | 高要求任务 | 低要求任务 |
|---|---|---|---|
| 准确性 | 是否能忠实抽取，不编造 | 结论判断、条款解释 | 候选召回 |
| 长上下文 | 是否能处理长标准文本 | 版本对比、整份标准摘要 | 单章摘要 |
| 推理能力 | 是否能处理跨标准关系 | 主题汇总、适用性判断 | 标签初筛 |
| 稳定性 | 是否输出格式稳定 | 自动入库前审核 | 草稿生成 |
| 速度 | 是否适合用户等待 | 即时问答 | 夜间批处理 |
| 成本 | 是否适合大批量 | 精加工少量关键文件 | 大批量预处理 |
| 可校验性 | 是否能给出来源片段 | 所有事实型任务 | 创意型问题生成 |

## 10. 任务分工总表

| 阶段 | 任务 | 推荐模型 |
|---|---|---|
| 架构设计 | 目录、Schema、页面模板 | 可靠即时高推理模型 |
| PDF/OCR 转文字 | 文本提取、OCR | 工具完成，不优先使用 LLM |
| 元数据抽取 | 标准号、标题、发布日期、实施日期 | 可靠即时中等能力模型初抽，后台模型投票，可靠即时高推理模型抽查 |
| 章节目录抽取 | 章节号、标题、摘要 | 可靠即时中等能力模型或不稳定中等能力后台模型 |
| 规范性引用文件抽取 | 引用标准列表 | 可靠即时中等能力模型 / 不稳定中等能力后台模型 + 规则校验 |
| 主题候选召回 | 找座椅、制动、灯光等相关条款 | 可靠即时中等能力模型 / 不稳定中等能力后台模型 |
| 条款解释 | 白话解释、适用对象、例外条件 | 可靠即时高推理模型 |
| 跨标准主题页 | 如座椅相关标准汇总 | 可靠即时高推理模型，不稳定高能力后台模型可先出草稿 |
| 版本对比 | 新旧版差异、影响分析 | 可靠即时高推理模型，不稳定高能力后台模型可先出草稿 |
| 时间线判断 | 现行版本、替代关系、过渡期 | 可靠即时高推理模型 |
| 高频问答沉淀 | 复杂问题答案页 | 可靠即时高推理模型 |
| 批量索引初稿 | 索引表、候选主题映射 | 可靠即时中等能力模型 / 后台模型 |
| 最终面向用户问答 | 综合回答、引用依据、不确定性说明 | 可靠即时高推理模型 |

## 11. 入库规则

### 11.1 不直接入库的内容

以下内容不得由中等能力模型或后台模型直接写入正式知识库：

- 法规或标准适用性结论；
- 是否存在过渡期的最终判断；
- 新旧版标准差异的最终结论；
- 跨标准主题的最终范围判断；
- 未带来源的事实陈述。

### 11.2 可以进入草稿区的内容

以下内容可以进入 `_drafts/`、`_candidates/` 或 `_reviews/`：

- 候选条款；
- 候选主题标签；
- 元数据初抽；
- 多模型投票结果；
- 标准页面初稿；
- 版本对比初稿；
- 高频问题候选。

### 11.3 正式入库要求

进入正式目录前，应至少满足：

- 有来源文件；
- 有条款号或章节号；
- 有原文片段或可回溯路径；
- 已经可靠即时高推理模型或人工审核；
- 不确定项已标记为 `unknown` 或写入 `_meta/unresolved-issues.md`。

## 12. 总结

推荐的总体策略是：

```text
不稳定后台模型：批量读资料、生成草稿、召回候选、做交叉检查
可靠即时中等能力模型：快速补充抽取、即时轻量任务、简单索引更新
可靠即时高推理模型：审核、判断、归纳、对比、最终问答
```

最终原则：

> 后台模型产物永远不直接进入正式知识库；所有事实必须能追溯到原文；最终结论由可靠即时高推理模型或人工审核。
